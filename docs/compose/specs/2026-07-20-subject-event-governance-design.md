# 对象、事件、验证与治理功能设计规格

## [S1] 目标与术语

UniBlack 记录的核心是一个“对象”及其关联账号和事件，不是孤立的案件。后续用户界面、API 文案、导出包和控制台统一使用以下术语：

- **对象（Subject）**：被记录的个人或账号实体。
- **账号（Account）**：对象在某平台上的可识别账号。
- **事件（Event）**：与对象有关的一次具体经过。新功能不再使用“案件”。
- **证据（Evidence）**：支撑事件的上传文件、文本文件或外部链接。
- **提交（Submission）**：登录用户创建对象、账号、事件及证据的操作。
- **申诉（Appeal）**：对具体事件提出的更正请求。
- **处罚（Sanction）**：对恶意或明显失实提交者的治理措施。

本规格将现有对外 Case 概念迁移为 Event。现有数据库/接口的兼容迁移必须独立设计，不能在本设计阶段直接删除历史表或数据。

## [S2] 对象身份与公开 ID

每个对象同时拥有：

- 内部主键：数据库 UUID，仅用于关系和内部操作。
- 公开 ID：系统生成、不可编辑的 `UBS_<ULID>`，例如 `UBS_01J9A7M4K8YF6V2Q3R5S8T1W0X`。

公开 ID 用于 URL、导出包名、证据文件前缀和人工沟通。服务端生成后检查数据库唯一约束；不接受客户端填写，也不使用可猜测递增编号。

对象字段：

| 字段 | 规则 |
| --- | --- |
| `display_name` | 提交者可填的通用名；为空时取第一条账号 `username` |
| `public_id` | `UBS_<ULID>`，全局唯一、不可修改 |
| `status` | `published`、`corrected`、`withdrawn`、`archived` |
| `created_by` / `created_at` | 提交者与创建时间，供治理和审计使用 |
| `profile_attributes` | 有限 JSONB 扩展字段；不得放入无约束的大型正文或文件内容 |

若第一条账号没有 `username`，提交页必须要求填写通用名，不能以账号 ID 或空字符串生成显示名。

## [S3] 账号模型与去重

对象可绑定任意数量账号。每条账号包含：

| 字段 | 说明 |
| --- | --- |
| `platform` | 标准平台 slug 或经规范化的自定义平台 slug |
| `platform_label` | 自定义平台对用户显示的名称 |
| `account_type` | `username`、`id`、`both` 或 `custom` |
| `username` | 平台昵称/用户名，可为空 |
| `account_id` | 平台稳定 ID，可为空 |
| `custom_attributes` | 小型 JSONB 键值扩展，值仅限字符串、数字、布尔值 |
| `is_primary` | 对象默认展示账号 |

唯一性优先以规范化的 `(platform, account_id)` 判定；没有 account ID 时使用 `(platform, username)`。规范化移除首尾空白、使用小写 platform slug；是否大小写敏感由平台配置声明。冲突不自动合并对象，而是提示提交者选择已有对象或创建管理员可复核的关联请求。

## [S4] 事件与默认发布

一个对象有多个事件。事件字段包括：标题、发生时间或时间区间、详情、严重度、可见状态、提交者、修订记录和证据清单。

登录且完成验证的用户提交后，系统在同一事务中创建对象（必要时）、账号、事件和提交审计，事件初始状态为 `published`。默认发布不是无治理：

- 服务端先验证必填字段、账号冲突、文本上限、文件类型/大小、演示人机令牌、邮箱验证码与提交处罚状态。
- `draft` 仅对提交者可见，不计入公开对象统计。
- `published` 对公众可见。
- `corrected` 保留公开修正说明与历史版本。
- `withdrawn` 不再按普通查询展示，但保留最小更正记录、申诉结论和审计。
- 管理员紧急隐藏使用 `hidden_pending_review`，必须填写原因并产生审计。

提交者可在短暂编辑窗口内更正草稿；发布后的事实修改走修订版本，不覆盖旧版本。

## [S5] 证据与可移植 JSON 包

证据统一为文件、文本或链接三类条目。数据库保存用于查询、权限、完整性和治理的索引，不把 JSON 文件当作唯一数据库：

- 证据元数据：对象 public ID、事件 ID、类型、原始文件名、存储键、MIME、大小、SHA-256、上传者、创建时间和可见性。
- 文本证据：服务端以 UTF-8 写入 `.txt`；最大 **200 KiB**，拒绝二进制和超限内容。
- 文件证据：允许列表和大小上限由站点配置控制；保存时保留原始文件名为展示字段，不能用它作对象存储键。
- 链接证据：保存 URL、标题、摘要和可选访问时间；不由后端擅自抓取内容。

每个对象导出为一个独立可移植包：

```text
subjects/
  UBS_<ULID>/
    manifest.json
    README.txt
    evidence/
      UBS_<ULID>_E001_F001.pdf
      UBS_<ULID>_E001_T001.txt
```

文件名格式为 `对象公开ID_事件序号_证据序号.后缀`。因此同一对象多个事件不会覆盖文件。导出时生成 `manifest.json`，其 `schema_version` 固定为 `1`，包含对象、账号、事件、证据文件名、SHA-256、原始文件名、导出时间和软件版本；不包含密码、邮箱验证码、captcha secret、JWT 或内部用户敏感信息。

`README.txt` 必须用中文说明目录结构、字段、SHA-256 校验、文本编码、导入约束和隐私边界。导入先做 schema 校验、哈希校验、public ID 冲突策略和预览，默认不覆盖既有对象。

## [S6] 申诉、更正与分级处罚

申诉针对单个事件，提交者可提供说明和补充证据。管理员结论为：

- `upheld`：维持事件。
- `corrected`：修正事件并说明变更。
- `withdrawn`：撤销事件并公开保留撤销原因。
- `malicious_submission`：确认提交存在恶意伪造、骚扰或重复滥用。

确认 `malicious_submission` 后，admin 可创建处罚：

| 类型 | 效果 |
| --- | --- |
| `warning` | 记录警告，不阻止提交 |
| `submission_suspension` | 指定到期时间前禁止创建/发布对象、事件和证据 |
| `submission_ban` | 永久禁止提交，仍可登录查看自己的处罚与申诉 |

处罚必须保存原因、关联事件/申诉、证据说明、执行人、开始/结束时间、撤销人/撤销原因与审计日志。处罚不自动删除用户、不抹除提交历史；被处罚用户可以对处罚提出一次申诉。自动化仅检查有效处罚，不自动决定“恶意”。

## [S7] 演示人机验证与邮箱验证

注册和发布提交均要求邮箱验证和人机验证。

### 演示人机验证

- 本项目发布的是内置演示 provider：界面为可访问的“我不是自动程序”确认卡，服务端签发短期、单用途、绑定 purpose/session 的演示 token。
- 项目不加载第三方 captcha JavaScript，不调用 Turnstile、reCAPTCHA 或 hCaptcha 验证端点。
- 控制台保留 provider 名称、Site Key、Secret、回调地址、开关和配置状态，以形成真实接入的配置契约；这些字段在本项目中不会触发真实网络验证。
- 设置文案必须显示“仅保存接入配置，本项目运行演示验证”，不得误称已接入真实 Provider。

### 邮箱验证

- 开发环境：当 `APP_ENV=development`，仅接受固定验证码 `123456`；不发送邮件，也不写入随机真实码。
- 生产环境：必须配置 SMTP；生成随机、单次使用、10 分钟有效验证码。未配置 SMTP 时验证请求失败，不能退回日志验证码。
- purpose 必须区分 `register`、`submission`、`appeal`；同一邮箱和 purpose 有发送频率限制，验证码不可跨 purpose 使用。
- SMTP 凭据仅存敏感 OptionMap 项并在 admin API 中脱敏；配置层继续遵循 env 默认值、内存缓存、DB 覆盖、控制台热更新的顺序。

## [S8] 提交页面体验

未登录访问 `/submit` 时显示提交说明，主发布按钮、上传区和草稿保存区为灰色禁用状态，并提供“登录后提交”主操作；不伪装为可提交的表单。

已登录用户使用单页分段提交，不使用会丢失长内容的多页向导：

1. **对象**：通用名与公开 ID 生成说明。
2. **账号**：可添加、删除、排序账号；平台模板和自定义属性编辑器。
3. **事件**：可添加多个事件，填写发生时间、详情和严重度。
4. **证据**：文件、文本、链接以同一清单展示；上传时显示最终文件名、大小、类型、哈希状态与错误。
5. **验证与发布**：邮箱码、演示人机卡、公开发布声明和提交处罚提醒。

页面保留段落锚点、草稿、错误定位、最终确认和低性能设备可用的渐进上传。明确文案：“提交后默认公开；错误或恶意内容可被申诉、修正、撤销，并可能导致提交权限处罚。”

## [S9] 统一管理控制台

控制台维持单一 `/admin` Shell，采用既定数据控制台布局：

| 分组 | 页面 |
| --- | --- |
| 概览 | 待处理申诉、近期事件、处罚与导出状态 |
| 内容与治理 | 对象、事件、申诉、处罚、审计 |
| 用户与访问 | 用户、角色、访问名单 |
| 站点与配置 | 站点与品牌、注册与验证、邮件 SMTP、演示 captcha 配置、限速、导入导出 |

“外观管理”更名为“站点与品牌”，管理项目显示名、描述、Logo、主题色、页脚和对外说明。系统设置依旧由 Catalog schema 驱动：分组、字段类型、敏感性、公开性和启用条件由后端返回；关联的 SMTP 与 captcha 设置应以原子批量更新保存。

对象、事件、申诉、处罚和导出页面使用明确筛选、状态徽章、审计入口、危险操作确认和小屏摘要视图。管理 UI 只改善可发现性，后端依旧强制 admin/moderator 权限。

## [S10] API、导出和权限边界

计划新增的 API 按资源组织：

- `/api/subjects`、`/api/subjects/{public_id}/accounts`：对象和账号。
- `/api/subjects/{public_id}/events`、`/api/events/{id}`：事件与版本。
- `/api/events/{id}/evidence`：证据元数据、上传和文本提交。
- `/api/events/{id}/appeals`、`/api/appeals/{id}/resolution`：申诉与结论。
- `/api/admin/sanctions`：处罚管理。
- `/api/admin/exports`、`/api/admin/imports`：导出任务、预览与导入。
- `/api/verification/demo`、`/api/verification/email`：演示人机与邮箱验证，按 purpose 限制。

公开 API 只能读取 `published` 或已公开修正的对象/事件，证据下载必须再检查事件可见性。文件上传、文本证据、发布提交、处罚、导入和导出均必须审计。所有写接口采用 UUID 内部 ID 或公开 ID 的明确语义，不能混用。

## [S11] 迁移、测试与非目标

实现必须采用新的可回滚 SQL migration；先新增 public ID、账号字段、事件/处罚/导出相关结构，再迁移读取路径，最后再决定旧 Case API 的弃用窗口。不得 destructive drop 后直接重建已有生产数据。

必须验证：ULID 唯一性、默认显示名、账号去重、事务原子性、文本上限/哈希、导出包可校验、导入冲突预览、默认发布、申诉更正、处罚期限、开发固定验证码、生产 SMTP 失败闭环、演示 captcha 不发第三方网络请求、权限与审计。

明确不实现：真实第三方 captcha adapter、无约束任意 JSON、数据库只存对象 ID、无审核的物理删除、自动裁定恶意、导出包中含 secret、基于递增 ID 的公开编号。
